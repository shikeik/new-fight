二、程序化资源生成方案

这是最关键的部分——所有视觉元素都用代码生成。

2.1 角色生成

玩家角色：用基础几何体拼接成低多边形风格的格斗家。

```
身体：CylinderGeometry 圆柱体（躯干）+ SphereGeometry（头部）+ BoxGeometry（四肢）
材质：MeshStandardNodeMaterial + 程序化渐变色纹理
颜色：玩家（蓝青色金属质感）、敌人（红橙色）
```

生成流程：创建骨架（Skeleton）→ 每个骨骼关联几何体Mesh → 组合为SkinnedMesh。Three.js的SkinnedMesh API提供了完整的骨骼驱动顶点能力。角色面数约800~1200三角面，移动端GPU友好。

骨架结构（约15根骨骼）：

· Root → Spine → Chest → Neck → Head
· 左右上臂 → 左右前臂 → 左右手
· 左右大腿 → 左右小腿 → 左右脚

2.2 动画生成（程序化关键帧）

不依赖动画文件，用代码生成关键帧序列：

```javascript
// 创建动画片段（伪代码）
function createIdleAnimation(skeleton) {
    // 待机动画：呼吸效果，躯干微旋转，手臂轻微摆动
    // 通过周期性旋转Root和Spine实现
}

function createPunchAnimation(skeleton) {
    // 0.3秒攻击动画：手臂快速伸出，身体前倾
    // 骨骼旋转关键帧：0s→0.15s→0.3s
}
```

每根骨骼的旋转（Quaternion）关键帧构成一个AnimationClip，通过AnimationMixer播放。

计划动画：待机、移动、跳跃、轻拳、重拳、受击、倒地。

2.3 场景生成

地面与竞技场：

· 地面：PlaneGeometry + 程序化网格纹理（棋盘格或环形纹），用Canvas 2D动态绘制纹理
· 边界墙：半透明BoxGeometry + 线框模式

背景：远景山脉或城市剪影用ExtrudeGeometry或LatheGeometry程序化生成，结合多层渐变色天空球。

灯光：主方向光 + 补光 + 边缘光，PBR材质（Metalness/Roughness）自动响应。

2.4 音效生成（Web Audio API）

用Web Audio API的OscillatorNode程序化生成音效波形：

音效 实现
拳击命中 100Hz正弦波 + 快速衰减包络（0.05s）
跳跃 200Hz→400Hz滑音
受击 80Hz + 轻微失真
胜利/KO 三音符上升音阶
背景音乐 简易鼓点loop

三、游戏系统设计

3.1 游戏循环

标准的 requestAnimationFrame 驱动，每帧执行：

1. 输入处理 → 更新摇杆和按钮状态
2. 逻辑更新 → 角色状态机、移动、物理（重力/摩擦力）、AI行为
3. 碰撞检测 → 攻击判定框 vs 受击框、角色间碰撞
4. 动画更新 → AnimationMixer.update(deltaTime)
5. 渲染 → WebGPURenderer.render()

3.2 碰撞与判定系统

角色移动碰撞：

· 每个角色：AABB包围盒（宽1.2m、高2.5m、深0.8m）
· 移动时检测场景边界 + 对手碰撞（角色不能互相穿过）

攻击判定框：

· 攻击动画的特定时间窗口触发（如轻拳在0.1s~0.25s之间）
· 攻击框：从角色手部位置生成的球体或胶囊体
· 命中时计算伤害 + 击退方向 + 触发HitStop（约0.1秒画面冻结，格斗游戏打击感核心）

3.3 角色状态机

有限状态机，每帧根据条件和输入切换：

```
状态枚举：
- IDLE: 待机
- WALK: 移动
- JUMP: 空中（上升/下降）
- ATTACK_LIGHT: 轻攻击
- ATTACK_HEAVY: 重攻击
- HIT: 受击硬直
- KNOCKDOWN: 倒地
- GETUP: 起身
- WIN/LOSE: 胜负
```

3.4 AI系统

敌人AI基于状态机+简单决策：

· 追逐：与玩家保持1.5~3米距离
· 攻击决策：距离足够时随机出轻拳/重拳（约每秒1.5次尝试）
· 防御：20%概率格挡（减少50%伤害）

3.5 移动端虚拟摇杆

左侧：固定摇杆区域，输出归一化方向向量（带死区0.15）
右侧：攻击按钮（轻拳、重拳） + 跳跃按钮

使用 touchstart/touchmove/touchend 事件。

四、渲染与性能优化

4.1 WebGPU 渲染器初始化

```javascript
import { WebGPURenderer } from 'three/webgpu';
const renderer = new WebGPURenderer({ 
    antialias: true,
    powerPreference: 'high-performance' 
});
await renderer.init();
renderer.shadowMap.enabled = true;
```

4.2 移动端性能优化

· 面数控制：角色<1500三角面，总场景<5000三角面
· 阴影：使用单方向光投射简单阴影，或仅用环境遮蔽贴图模拟
· 粒子系统：命中特效用GPU粒子（compute shader可选，也可用简单Sprite）
· 纹理压缩：程序化纹理用Canvas生成，无需外部文件
· 暂停优化：游戏暂停或后台时停止 requestAnimationFrame
· 命令缓冲区：WebGPURenderer内部管理，无需手动干预

4.3 后处理与视觉反馈

用 RenderPass 链实现：

· 受击闪白：叠加全屏白色半透明quad
· 屏幕震动：命中时相机位置抖动
· 血条：HTML5 Canvas绘制的2D血条叠加在3D角色上方

五、完整游戏流程

1. 加载/初始化：生成程序化资源，显示"TAP TO START"
2. 选人/准备：单角色，直接进入战斗
3. 战斗：玩家 vs AI，100秒倒计时，双方各100HP

   · HP归零 → KO胜利

   · 显示胜利/失败文字 + "再战"按钮
5. 循环：点击再战重置状态

七、风险与应对

风险 可能性 应对方案
WebGPURenderer在特定安卓机型不兼容 低 Three.js自动fallback到WebGL 2.0
程序化动画不够自然 中 采用缓动函数（ease-in-out）让关键帧过渡更平滑
虚拟摇杆误触/响应延迟 中 固定摇杆区域，使用preventDefault防止页面滚动
移动端发热/掉帧 低 降级阴影质量，降低粒子数量，控制面数
