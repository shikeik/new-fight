# 代码质量规范指南

> 基于 terrain-lab 重构实战案例（1523行 → 433行，9批次重构）  
> 本文档以**正反例双标准**呈现，旨在建立可执行的代码质量标准

---

## 1. 引言

### 1.1 为什么需要这份规范

在 terrain-lab 模块的维护过程中，我们遭遇了典型的**技术债务爆发**：

- 原始代码 1523 行，混杂在单个 `main.ts` 文件中
- 改一处评估逻辑，需要在 4-6 个地方同步修改
- 添加新功能时，需要理解整个文件的上下文
- 测试无法单独测试业务逻辑，因为与 DOM 操作深度耦合

经过 9 批次系统性重构，最终降至 433 行，核心改进：

| 指标 | 重构前 | 重构后 | 改进 |
|------|--------|--------|------|
| 单文件行数 | 1523 | 433 | -72% |
| DOM操作散落点 | 61处 | 0处（全部收口） | 集中化 |
| 重复代码块 | 8处 | 0处 | 消除 |
| 纯逻辑函数占比 | 30% | 85% | 可测试性↑ |

### 1.2 核心原则

```
DRY  >  一切（不要重复自己）
SRP  >  函数长度（单一职责优于短函数）
显式 > 隐式（魔法数字必须具名）
纯函数 > 副作用（业务逻辑与副作用分离）
复用 > 新建（新增前先查已有实现）
抽离 > 复制（修改时遇重复立刻封装）
```

### 1.3 铁律声明

以下两条为**强制性规范**，违反即构成技术债务：

#### 铁律一：新增代码前必须查重

> **在编写任何新代码之前，必须先检索项目中是否已有类似实现。如有，必须复用；如不完全匹配，必须先封装为可复用形式再调用。**

**反面教材**：
- ❌ 看到需要随机选择样本，直接写 `Math.floor(Math.random() * dataset.length)`
- ✅ 先查是否有 `getRandomSample()` 或类似工具，没有则创建并复用

**执行标准**：
1. 写代码前，用关键词在项目中搜索（如 `random`、`gradient`、`updateMetrics`）
2. 发现相似实现 ≥ 70% 匹配度 → 直接复用
3. 发现相似实现 30%-70% 匹配度 → 提取公共部分，封装后复用
4. 完全无匹配 → 考虑该功能是否通用，通用则设计为可复用形式

#### 铁律二：修改时遇重复必须抽离

> **在修改任何代码逻辑时，必须主动搜索是否有相同逻辑存在于别处。一旦发现，**立即停止当前修改**，先将重复逻辑抽离封装，统一调用，再修改封装后的单一实现。**

**反面教材**：
- ❌ 发现 `trainSupervised` 和 `trainUnsupervised` 都需要修改评估逻辑，分别在两个函数里改
- ✅ 立即停止，提取 `evaluateAndReport()`，两处统一调用，只改封装函数

**执行标准**：
1. 准备修改功能 X 时，全局搜索与 X 相关的关键词
2. 发现 N 处有相同/相似逻辑 → **立即停止修改**
3. 创建封装函数 `extractCommonX()`
4. 将所有 N 处替换为 `extractCommonX()` 调用
5. 修改 `extractCommonX()` 的实现
6. 验证所有 N 处调用点行为一致

**绝对禁止**：
- 明知有重复，先改了再说，下次再重构
- 复制粘贴已有代码，改几行变成新功能
- 在 N 处重复代码上分别打补丁

## 2. DRY 原则 —— 重复代码是致命伤

**严重等级：🔴 最高**

### 2.1 问题本质

重复代码不仅仅是"多敲了几行"的问题，而是：

1. **维护地狱**：改一处逻辑，需要在 N 处同步修改
2. **遗漏风险**：必然有地方忘记修改，导致不一致
3. **阅读负担**：读者需要对比多份相似代码找差异
4. **测试膨胀**：同样的逻辑需要重复测试

### 2.2 正反例：课程学习的双模式重复

#### ❌ 坏代码（重构前）

```typescript
        const evaluation = calculateReward(action, isValid, action === optimal, UNSUPERVISED_CONFIG)

        accumulateGradients({ dEmbed: gEmbed, dW1: gW1, db1: gb1, dW2: gW2, db2: gb2 }, 
          state.net, sample.indices, evaluation, batchSize)
      }

      updateNetwork(state.net, { dEmbed: gEmbed, dW1: gW1, db1: gb1, dW2: gW2, db2: gb2 }, 1)
      state.trainSteps++

    const { accuracy, validRate, loss } = evaluateDataset(state.dataset, state.net)

    updateMetrics({ loss, acc: accuracy, validRate, epsilon: newEpsilon, progress: Math.min(state.trainSteps / maxTotalSteps, 1) * 100 })
```

- 两个函数 200+ 行，80% 结构相同
- 循环结构、评估逻辑、快照保存、进度更新完全重复

#### ✅ 好代码（重构后）

```typescript
// training-engine.ts - 纯逻辑，与UI无关
export class TrainingEngine {
  constructor(
    private state: AppState,
    private onStep: (result: TrainingResult & { progress: number }) => Promise<void>
  ) {}

  // 统一的阶段运行框架
  async runStage(
    targetAcc: number,
    maxSteps: number,
    trainStrategy: (batchSize: number) => Promise<void>
  ): Promise<boolean> {
    const { batchSize } = TRAIN_CONFIG
    const stepsPerBatch = 100
    
    while (this.state.trainSteps < maxSteps) {
      // 执行训练（策略模式，由调用方决定如何训练）
      for (let s = 0; s < stepsPerBatch; s++) {
        await trainStrategy(batchSize)
        this.state.trainSteps++
      }

      // 统一评估
      const result = this.evaluateDataset()
      const progress = Math.min(this.state.trainSteps / maxSteps, 1) * 100
      
      await this.onStep({ ...result, progress })
      
      if (result.accuracy >= targetAcc) return true
      await new Promise(r => setTimeout(r, 1))
    }
    return false
  }

  // 监督学习策略
  async trainSupervisedBatch(batchSize: number): Promise<void> {
    const buffer = createGradientBuffer()
    for (let b = 0; b < batchSize; b++) {
      const sample = this.getRandomSample()
      accumulateSupervisedGrad(buffer, this.state.net, sample.indices, sample.y, batchSize)
    }
    updateNetwork(this.state.net, buffer, 1)
  }

  // 无监督学习策略
  async trainUnsupervisedBatch(batchSize: number): Promise<void> {
    const buffer = createGradientBuffer()
    for (let b = 0; b < batchSize; b++) {
      const sample = this.getRandomSample()
      const action = this.selectAction(sample)
      const evaluation = this.evaluateAction(sample, action)
      accumulateGradients(buffer, this.state.net, sample.indices, evaluation, batchSize)
    }
    updateNetwork(this.state.net, buffer, 1)
  }
}

// curriculum-controller.ts - 协调层，只负责组装
export class CurriculumController {
  async runStage(targetAcc: number, maxSteps: number, mode: LearningMode): Promise<boolean> {
    const engine = new TrainingEngine(this.state, async (result) => {
      this.uiManager.updateMetrics(result)
      this.snapshotManager.addSnapshot()  // 快照逻辑收口
      await new Promise(r => setTimeout(r, 1))
    })

    // 根据模式选择训练策略
    const trainStrategy = mode === "supervised" 
      ? (bs: number) => engine.trainSupervisedBatch(bs)
      : (bs: number) => engine.trainUnsupervisedBatch(bs)

    return engine.runStage(targetAcc, maxSteps, trainStrategy)
  }
}
```

**改进要点**：
- 提取共同框架 `runStage()`，只暴露差异点（训练策略）
- 策略模式：通过函数参数注入不同的训练逻辑
- UI 更新通过回调函数解耦
- 快照、评估、进度更新全部收口到统一位置

### 2.3 检测重复代码的指标

```
⚠️ 危险信号：
- 复制粘贴后只改动了 3-5 行
- 两个函数结构相同，只有中间某段逻辑不同
- 修改某功能时需要在多个文件搜索替换
- 测试用例中有多份相似的 setup 代码
```

## 3. SRP 原则 —— 单一职责

**严重等级：🔴 高**

### 3.1 问题本质

一个模块/函数应该**只有一个修改的理由**。当业务逻辑、UI更新、数据处理混在一起时：

1. 修改 UI 可能影响业务逻辑
2. 无法单独测试业务逻辑
3. 代码阅读需要理解无关的上下文

## 4. 函数长度控制

## 9. 纯函数与副作用分离

**严重等级：🟠 中高**

### 9.1 原则

- 业务逻辑必须是纯函数（输入相同，输出相同，无副作用）
- 副作用（DOM操作、网络请求、随机数）集中在边界
- 便于测试：纯函数可以直接断言，副作用可以 mock

### 9.2 正反例

#### ❌ 坏代码

```typescript
// 业务逻辑与副作用混杂
async function trainStep() {
  const buffer = createGradientBuffer()
  
  // 副作用1：随机数生成（不可测试）
  const idx = Math.floor(Math.random() * state.dataset.length)
  const sample = state.dataset[idx]
  
  // 业务逻辑
  accumulateGradients(buffer, state.net, sample.indices, sample.y, batchSize)
  updateNetwork(state.net, buffer, 1)
  
  // 副作用2：状态修改
  state.trainSteps++
  
  // 副作用3：DOM更新
  document.getElementById("step-count")!.textContent = String(state.trainSteps)
  
  // 副作用4：延时
  await new Promise(r => setTimeout(r, 1))
}
```

#### ✅ 好代码

```typescript
// training-engine.ts
export class TrainingEngine {
  // 纯函数：给定相同输入，总是产生相同输出
  calculateGradients(sample: Sample, batchSize: number): GradientBuffer {
    const buffer = createGradientBuffer()
    accumulateGradients(buffer, this.state.net, sample.indices, sample.y, batchSize)
    return buffer
  }

  // 副作用集中在协调方法
  async trainBatch(getRandomSample: () => Sample): Promise<TrainingResult> {
    // 副作用：获取随机样本
    const sample = getRandomSample()
    
    // 纯逻辑
    const gradients = this.calculateGradients(sample, this.config.batchSize)
    
    // 副作用：更新网络
    updateNetwork(this.state.net, gradients, 1)
    this.state.trainSteps++
    
    // 副作用通过回调通知
    await this.onStep({ step: this.state.trainSteps })
    
    return this.evaluate()
  }
}
```

## 12. 重构检查清单

在提交代码前，对照以下清单自检：

### 12.1 DRY 检查

- [ ] 没有复制粘贴后只改几行的代码块
- [ ] 相似的逻辑已提取到共用函数
- [ ] 配置没有散落在业务代码中
- [ ] DOM 操作没有重复出现在多个函数中

### 12.2 SRP 检查

- [ ] 每个函数只做一件事
- [ ] 函数名称能完整描述其功能
- [ ] 业务逻辑与 UI 操作分离
- [ ] 副作用与纯逻辑分离

### 12.3 长度检查

- [ ] 函数长度 < 50 行（理想）或 < 100 行（上限）
- [ ] 文件长度 < 500 行
- [ ] 类的方法数 < 15 个

### 12.4 命名检查

- [ ] 所有标识符都有明确的含义
- [ ] 布尔变量使用 is/has/should 前缀
- [ ] 常量使用 UPPER_SNAKE_CASE
- [ ] 日志标签统一格式 `"TAG"`（无方括号）

### 12.5 类型检查

- [ ] 没有 `any` 类型
- [ ] 函数参数有类型声明
- [ ] 函数返回值有类型声明
- [ ] 使用了具体的联合类型替代通用类型

### 12.6 模块边界检查

- [ ] 依赖方向正确（上层依赖下层）
- [ ] 使用了依赖注入
- [ ] 模块之间通过接口交互

## 13. 重构流程建议

当需要修改旧代码时，遵循以下步骤：

### 13.1 识别坏味道

```
1. 找到需要修改的功能点
2. 搜索是否有相似的代码存在于别处
3. 如果存在重复 → 先提取共用函数
4. 如果函数过长 → 先拆分为小函数
5. 如果混杂职责 → 先分离纯逻辑与副作用
```

### 13.2 渐进式重构

```
批次1: 提取纯工具函数（如 Canvas DPR 计算）
批次2: 提取绘制逻辑到 renderer.ts
批次3: 提取 DOM 操作到 ui-manager.ts
批次4: 提取业务逻辑到 training-engine.ts
批次5: 提取状态管理到各 Manager
...
每批次：保持测试通过，小步提交
```

### 13.3 重构前问自己

1. 我要修改的逻辑在其他地方也有吗？
2. 如果以后要改这个逻辑，需要改几处？
3. 这个函数能否在不看其他代码的情况下理解？
4. 这个函数能被单元测试吗？

## 14. 总结

### 14.1 核心原则优先级

```
        ┌─────────────────────────┐
        │   查重与复用铁律         │  ← 最高优先级
        │ （新增必查、修改必抽离）  │
        └────────┬────────────────┘
                 ▼
        ┌─────────────────┐
        │   DRY 原则      │  ← 不要重复自己
        │  （消除重复）    │
        └────────┬────────┘
                 ▼
        ┌─────────────────┐
        │   SRP 原则      │  ← 每个模块只做一件事
        │  （单一职责）    │
        └────────┬────────┘
                 ▼
        ┌─────────────────┐
        │   纯函数优先     │  ← 业务逻辑无副作用
        │  （可测试性）    │
        └────────┬────────┘
                 ▼
        ┌─────────────────┐
        │   显式优于隐式   │  ← 魔法数字必须具名
        │  （命名与配置）   │
        └─────────────────┘
```

### 14.2 关键指标

| 指标 | 危险值 | 目标值 |
|------|--------|--------|
| **新增代码未查重** | 发生 | **0 次** |
| **修改时重复未抽离** | 发生 | **0 次** |
| 单文件行数 | > 1000 | < 500 |
| 函数长度 | > 100 行 | < 50 行 |
| DOM操作散落点 | > 20 处 | 0 处（全部收口） |
| 重复代码块 | > 0 | 0 |
| any 类型使用 | > 0 | 0 |

### 14.3 每日三省

编写或修改代码前，问自己：

1. **查重了吗？** —— 我要写的功能，项目中是否已有实现？
2. **能复用吗？** —— 如果有，我是否直接复用或封装后复用？
3. **要抽离吗？** —— 如果修改时发现重复，我是否立即停止并抽离封装？

**警示语**：
